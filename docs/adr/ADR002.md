# ADR-002 — Estratégia de Armazenamento

**Status:**  Aceito  |  **Data:** [2026-04-03]  |  **Autores:** [Carlos Tomasi, Gustavo Antonius, Leonardo Latzke e Renan Apolináro]

## Contexto

A plataforma VendaMais necessita armazenar dados provenientes do processo de ingestão de forma eficiente, garantindo confiabilidade, escalabilidade e suporte a análises no Power BI.

Os dados coletados do ERP passam inicialmente por uma etapa de armazenamento em formato bruto, sendo posteriormente transformados e estruturados para consultas analíticas.

A solução deve suportar crescimento no volume de dados ao longo do tempo, mantendo baixo custo para armazenamento inicial e alto desempenho para consultas.

Além disso, é necessário garantir fácil integração com os serviços utilizados na arquitetura, bem como baixo esforço de manutenção e rápida implementação.## Decisão

Foi definida a utilização do Azure SQL Database como principal tecnologia para armazenamento dos dados estruturados da aplicação.

O banco de dados será responsável por armazenar os dados já organizados em formato relacional, permitindo consultas eficientes e integração direta com ferramentas de Business Intelligence.

A modelagem seguirá o padrão relacional, com foco em otimização para leitura (consultas analíticas), garantindo desempenho adequado para uso no Power BI.

Essa decisão contempla exclusivamente a camada de armazenamento estruturado, não incluindo armazenamento de dados brutos ou não estruturados.

Foi definida uma estratégia de armazenamento em duas camadas:

Azure Blob Storage será utilizado para armazenamento de dados brutos, permitindo baixo custo, alta escalabilidade e flexibilidade no armazenamento de arquivos.
Azure SQL Database será utilizado para armazenamento estruturado, otimizado para consultas analíticas e integração com o Power BI.

O fluxo de dados seguirá o modelo:

Ingestão > Blob Storage > Transformação > Azure SQL Database > Power BI

Essa abordagem separa dados brutos de dados processados, permitindo melhor organização, reprocessamento de dados e maior eficiência nas consultas analíticas.

O escopo desta decisão abrange toda a estratégia de armazenamento da solução, incluindo dados brutos e estruturados.

## Consequências

(+) Separação entre dados brutos e estruturados
Permite melhor organização e governança dos dados.

(+) Redução de custos de armazenamento inicial
Blob Storage oferece baixo custo para grandes volumes de dados.

(+) Alta performance em consultas analíticas
Azure SQL Database é otimizado para leitura e integração com BI.

(+) Flexibilidade para reprocessamento de dados
Dados brutos podem ser reutilizados para novas transformações.

(+) Escalabilidade da solução
Cada camada pode crescer de forma independente.

(-) Maior complexidade arquitetural
Necessidade de gerenciar duas camadas de armazenamento.

(-) Dependência de pipeline de transformação
É necessário um processo adicional para mover e estruturar os dados.

(-) Latência adicional no fluxo de dados
O armazenamento intermediário pode aumentar o tempo até a disponibilização final.

## Alternativas rejeitadas

Armazenar diretamente no Azure SQL Database;
Foi rejeitada devido ao maior custo para grandes volumes e menor flexibilidade para dados brutos.

Utilizar apenas Azure Data Lake;
Foi rejeitada por maior complexidade e por não ser necessário para o volume e escopo atual do projeto.

Utilizar banco NoSQL;
Foi rejeitado por não atender adequadamente às necessidades de consultas relacionais e integração com ferramentas de BI.
